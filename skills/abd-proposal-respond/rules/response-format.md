# Response Format — Common Instructions

Generalized guidance for proposal responses. Adapt per RFP; add project-specific DO/DO NOT to strategy.

## Structure

**DO** — Use consistent structure per question: lead paragraph → bulleted list with **Bold labels** → source references. Include time horizons (e.g. near-term and beyond) when the question spans them.

**DON'T** — Do not mix formats (table vs block) within the same section. Do not omit the lead paragraph or use bare bullet lists without context.

## Lead Paragraph

**DO** — Articulate outcomes in the client's language. Be aspirational and specific. Use "our firm would aim to accomplish [concrete outcomes]" and "Key to this would include [context]." Then: "We will achieve these outcomes through:"

**DON'T** — Do not restate the question generically. Do not use a robotic list of topic words. Do not skip the aspirational framing.

## Bullet Prose

**DO** — Explain the *method* and *mechanism*. Use varied sentence structure. Show practitioner reasoning. Each bullet = 2–3 substantive sentences.

**DON'T** — Do not use the formula "We will [verb] [thing]... to achieve [outcome]" for every bullet. Do not repeat "We will" at the start of every sentence. Do not give one shallow sentence per bullet. Do not just restate the outcome—explain *how* we get there.

## Bullet Labels

**DO** — Use *approach* or *method* names: "OKR alignment," "Change Canvas," "Stabilized flow." These name what we do, not the output.

**DON'T** — Do not use output/metric names as labels. Do not use vague labels. Prefer accelerator or method names from source materials when they fit.

## Source References

**DO** — Use a consistent format per RFP (e.g. `*See Appendix X (Full Name)*` or inline citations). One reference per bullet unless both are genuinely needed.

**DON'T** — Do not tack on extra source names after the main reference. Do not mix appendix refs with inline citations in the same bullet unless the RFP format allows it.

## Accelerators (Appendix References)

**DO** — When referencing an accelerator, define it (appendix letter, framework name) and accumulate it in the **Accelerator Table** (e.g. `Accelerator Table.md`). Add or update a row with: Appendix letter, Framework Name, Slide File, Slide Numbers, URL. Keep URLs in the table.

**DO** — Use `*See Appendix X (Full Name)*` at the end of bullets when the approach warrants a slide appendix.

**DON'T** — Do not reference an appendix without adding it to the Accelerator Table. Do not omit the URL when available from memory or source links.

## Tailoring to the Question

**DO** — Connect each bullet to the specific question. Use the question's language and intent. Tie methods to *what the question asks* (e.g. success → achieving, not just tracking; measurement → what we observe).

**DON'T** — Do not list practices generically. Avoid bullets that could apply to any question. Each bullet must answer *this* question, not a generic one.
